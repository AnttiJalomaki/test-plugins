## Toolchain installation and bootstrap

### Relocating the `go` command

`GOROOT_FINAL` no longer affects an installed toolchain. A distribution that places `go` somewhere other than `$GOROOT/bin/go` must install a symlink instead of relocating or copying the binary.

### Bootstrap floors

Building Go 1.24 from source requires Go 1.22.6 or later. Building Go 1.26 from source requires Go 1.24.6 or later.

## Cgo contracts and linker flags

### Large C-linker argument lists

`cmd/cgo` accepts `-ldflags` for C-linker arguments. The `go` command uses it automatically, preventing very large `CGO_LDFLAGS` values from overflowing the process argument list.

### Call-contract annotations

`#cgo noescape f` promises that pointers passed to C function `f` do not escape. `#cgo nocallback f` promises that `f` never calls back into Go. These promises enable cheaper calls and must reflect actual C behavior.

Receiver methods whose receiver type is generated by cgo are rejected even when an alias hides the receiver type.

### Immediate ELF binding

Pass linker flag `-bindnow` to enable immediate function binding for dynamically linked ELF executables, including PIE binaries.

```sh
go build -ldflags=-bindnow ./...
```

## Build identifiers and executable layout

### Native identifiers

The linker emits a GNU build ID on ELF and a UUID on Mach-O by default, derived from the Go build ID. Pass `-B none` to suppress it or `-B 0xNNNN` to provide a hexadecimal value.

### Debug information

The compiler and linker emit DWARF 5 by default. Use `GOEXPERIMENT=nodwarf5` temporarily when an older debugger or analyzer cannot read it.

### Linker and analyzer changes

On `windows/arm64`, cgo programs can request internal linking with `-ldflags=-linkmode=internal`.

Executable analyzers and custom linker scripts must account for all of these layout changes:

- A new `.go.module` section.
- A relocation-free `.gopclntab` whose `pcHeader` text address is zero.
- Moved funcdata and findfunctab data.
- Removal of `.gosymtab`.
- Address-sorted sections in internally linked ELF executables.

## Architecture profiles and SIMD

### Arm64 and RISC-V target profiles

`GOARM64` accepts `v8.0` through `v8.9` or `v9.0` through `v9.5`, with an optional `,lse` or `,crypto` suffix. Its default is `v8.0`.

`GORISCV64` initially accepts `rva20u64` or `rva22u64` and defaults to `rva20u64`; `GORISCV64=rva23u64` is also available in Go 1.25.

```sh
GOOS=linux GOARCH=arm64 GOARM64=v8.2,lse go build ./...
GOOS=linux GOARCH=riscv64 GORISCV64=rva22u64 go build ./...
```

### Floating-point fusion

At `GOAMD64=v3` or higher, fused operations may change floating-point results. Write `float64(a*b)+c` when fusion must be prevented.

### Experimental architecture-specific SIMD

`GOEXPERIMENT=simd` exposes the unstable, non-portable `simd/archsimd` package on amd64. It supplies 128-, 256-, and 512-bit vector types with architecture-specific operations and is distinct from a future portable SIMD API.

## WASI and WebAssembly

### Runner requirement

`GOROOT/misc/wasm/go_wasip1_wasm_exec` requires Wasmtime 14.0.0 or newer.

### Exportable WASI reactors

For `GOOS=wasip1 GOARCH=wasm`, `-buildmode=c-shared` builds a reactor or library rather than a command. Use `//go:wasmexport` to export Go functions to the host.

WebAssembly imports and exports may use `bool`, `string`, `uintptr`, and permitted pointer types in addition to numeric types and `unsafe.Pointer`.

### Instruction floor

WebAssembly targets require sign-extension and non-trapping float-conversion instructions. The `GOWASM` settings `signext` and `satconv` are ignored.

## Native process handles

`os.Process.WithHandle` provides controlled access to a process pidfd on Linux 5.4 or later, or to its Handle on Windows.

## Operating-system and port transitions

### macOS and Windows

Go 1.24 is the final release supporting macOS 11. Go 1.25 requires macOS 12, and Go 1.26 is the final release supporting macOS 12.

The 32-bit `windows/arm` port is marked broken in Go 1.24, and Go 1.25 is the final release containing it.

### Linux and BSD

Linux targets require kernel 3.2 or later. The `freebsd/riscv64` port is broken in Go 1.26.

### Architecture ports

`linux/loong64` gains race-detector support, cgo traceback, and internal linking in Go 1.25, while RISC-V gains plugin support. In Go 1.26, `linux/riscv64` gains race-detector support, s390x passes function arguments and results in registers, and support for the ELFv1 ABI on `linux/ppc64` reaches its final release.

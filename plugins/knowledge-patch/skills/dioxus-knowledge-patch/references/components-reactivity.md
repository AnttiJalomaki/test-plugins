# Components, Reactivity, Events, and Hooks

## Component API and properties

### Scope-free components and copyable signals

Since 0.5.0, components take properties directly instead of receiving `cx: Scope`; the bump lifetime is gone, `Element` is static, and runtime functions such as `spawn` and `consume_context` can be called without a scope. `use_signal` returns a `Copy` handle even when its value is not `Copy`; `use_signal_sync` produces a `Send + Sync` handle for cross-thread use.

```rust
fn Counter() -> Element {
    let mut count = use_signal(|| 0);
    spawn(async move { println!("{}", count()) });
    rsx! { button { onclick: move |_| count += 1, "{count}" } }
}
```

Properties are memoized and normally implement `Clone + PartialEq`. Historically, a non-clone value could be declared behind `ReadOnlySignal` and the props machinery handled the wrapper; current code should prefer the readable interfaces because `ReadOnlySignal` is deprecated.

### Reactive property boundaries

Declare readable component props as `ReadSignal<T>`. RSX automatically decays signals, memos, and store lenses into it and can lift a plain `T`, so derived hooks remain reactive without coupling the component to one storage primitive.

```rust
#[component]
fn Validator(is_valid: ReadSignal<bool>) -> Element {
    rsx! { if !is_valid() { "Invalid" } }
}

rsx! { Validator { is_valid: true } }
```

An `Option<T>` prop is optional at the call site unless annotated `#[props(!optional)]`. `#[props(into)]` accepts values convertible through `Into`, and `String` props accept formatted-string syntax. Constructed properties may be spread with `..props`; explicit fields appearing in the component invocation override spread fields.

Generated generic props stopped receiving unnecessary `Clone` bounds in 0.7.6. Props generated by `#[component]` are non-exhaustive in 0.8.0-alpha.0, so downstream code must not exhaustively match or directly construct them on the assumption that no fields can be added.

### Hook order

Hooks must stay out of conditionals, loops, and closures. Dioxus does permit an early return between hook calls so suspense and error `?` propagation can work, though placing all hooks before the return keeps the effective hook list stable and is easier to audit.

## Signals, subscriptions, and context

### Subscription rules

A component subscribes when it reads a signal during rendering, not when the only read occurs in an event handler or async task. Hooks such as `use_memo` and `use_future` track signal reads made inside them. `peek()` performs a non-subscribing read, while `use_reactive` turns a tuple of otherwise untracked dependencies into a tracked closure.

```rust
static COUNT: GlobalSignal<i32> = Signal::global(|| 0);

fn App() -> Element {
    let half = use_memo(move || COUNT() / 2);
    rsx! { "{half}" }
}
```

`GlobalSignal` replaced Fermi-style atoms, and ordinary signals can be shared through `use_context_provider` and `use_context`. `consume_context` also works in event handlers and polled futures because they run with the current runtime. Calling `provide_context` dynamically replaces the current component's value, but that replacement is not reactive.

A global static is scoped to one app instance: SSR tenants and Desktop windows receive independent values. `Memo::global` provides app-local derived global state.

### Batching, memos, and effects

Writes in one runtime step are batched and their effects deduplicated. An `.await` is a step boundary, so pending UI can paint before the future resumes. Reading a dependent memo immediately after a write recomputes it synchronously.

A memo tracks inputs but publishes only when `PartialEq` says its output changed, insulating consumers from irrelevant churn. Effects run after the UI paints, so external state is updated only for a component that survived rendering.

### Ownership and disposal

Signals created by a hook or `Signal::new()` belong to the current component and are disposed on unmount; reading a copied handle afterward panics. Repeatedly putting fresh `Signal::new()` values in another collection retains their contents until that owner unmounts unless they are manually disposed. Prefer store or reactive-collection mapping for dynamic items.

## Stores and lenses

`#[derive(Store)]` generates projections into nested fields, and reactive collection implementations invalidate only affected items rather than every collection consumer (since 0.7.0).

```rust
#[derive(Store)]
struct Dir {
    children: BTreeMap<String, Dir>,
}

let children = directory.children();
```

Generated projections have the conceptual type `Store<Value, Lens>`, where `Lens` is deliberately unnamed. Generic helpers should accept `impl Readable<Target = Value>` or `impl Writable<Target = Value>`. At component boundaries, writable and read-only lenses are boxed and decay to `ReadSignal` or `ReadStore` props.

```rust
fn clear_title(state: Store<HeaderState, impl Writable<Target = HeaderState>>) {
    state.title().take();
}
```

For an `Option` or `Result` lens, `.transpose()` yields an option/result containing the inner lens and preserves precise reactivity. `.as_ref()` performs a read and cannot be lensed farther. An impl annotated `#[store]` becomes an extension trait for every lens; `&self` methods gain `Readable` bounds and `&mut self` methods gain `Writable` bounds.

```rust
let subtitle: Option<Store<String, _>> = header.subtitle().transpose();

#[store]
impl<Lens> Store<UserData, Lens> {
    fn clear_name(&mut self) {
        self.name().take();
    }
}
```

`dioxus-stores` added `SyncStore` and `use_store_sync` for synchronized stores in 0.7.2. The signals API added `WritableResultExt` for writable `Result` values in 0.7.4. Preview store derives preserve original member visibility through visibility witness types in 0.8.0-alpha.0; a generated accessor cannot bypass Rust visibility.

Iteration through `Signal<Vec<T>>` cannot lend values borrowed from its read guard to child props. Clone cheap values, pass the collection and an index, or use a `Store`, whose iterator yields per-item lenses that decay into readable props.

```rust
let items = use_store(|| vec![1, 2, 3]);

rsx! {
    for count in items.iter() {
        CountRow { count }
    }
}
```

## Actions and task ownership

`use_action` combines a task with stored `Result` state. `.call(input)` starts it, `.value()` yields `None` or the latest result, and a new call while pending cancels and replaces the old task. This avoids stale-result races that repeated async event handlers can create.

```rust
let mut double = use_action(|value: i32| async move {
    anyhow::Ok(value * 2)
});

double.call(21);
if let Some(Ok(value)) = double.value() {
    println!("{value}");
}
```

`spawn` attaches a single-threaded task to the current component and cancels it on unmount. Keep its `Task` to call `.cancel()`. `spawn_forever` attaches genuinely app-long work to the root; a root-owned future must not retain signals owned by a shorter-lived component. The executor accepts `!Send` futures, but CPU-bound work belongs on a native thread or Web Worker.

```rust
let task = spawn(async move { run_job().await });
task.cancel();

use_hook(|| spawn_forever(async move { run_background_sync().await }));
```

`Action` implements `PartialEq` as of 0.7.6.

## Resources, suspense, and errors

### Result-based elements and error boundaries

The older 0.5.0 transition introduced `.throw()` on a `Result<_, E: Debug>` to turn failures into render error state for an ancestor boundary. Since 0.6.0, `Element` itself is a `Result`, `RenderError` accepts ordinary errors, and components and async event handlers can use `?` directly. Render an empty `rsx! {}` instead of the removed `VNode::None`.

```rust
fn Slider() -> Element {
    let value: f64 = "1234".parse()?;
    rsx! { "{value}" }
}
```

An `ErrorBoundary` handles errors through `handle_error` and can be nested to create recovery scopes. Fullstack SSR status handling has additional requirements described in [fullstack-routing.md](fullstack-routing.md).

### Resources and loaders

`use_resource` tracks reactive reads in its future, cancels and restarts on dependency changes, and notifies readers after every run even when the returned value compares equal. Its output must be cloneable; `CapturedError` wraps `anyhow::Error` in an `Arc`, and `dioxus::Ok(value)` conveniently constructs `Result<T, CapturedError>`.

For futures returning `Result<T, E>`, `use_loader` yields `Result<Loader<T>, Loading>`, allowing `?` to route pending and error states to suspense and error boundaries. Start all independent loaders before the first `?` to prevent request waterfalls.

```rust
let breeds = use_loader(move || async move {
    reqwest::get("https://example.test/breeds")
        .await?
        .json::<Breeds>()
        .await
})?;
```

### Suspense

`SuspenseBoundary` renders a fallback while a descendant resource is pending; a child can call `.suspend()?` and may use `with_loading_placeholder` to propose a task-specific view. Suspense also operates in native apps. Fullstack suspense uses `use_server_future`, which serializes the result for hydration and suspends automatically; only its outer closure is reactive, so dependency reads must occur there.

## Events, callbacks, and mounted elements

Events expose portable data and renderer-native data. The 0.5.0 Web API exposed `.web_event()`; current Web code can use `event.downcast::<web_sys::Event>()` when the portable trait is insufficient. `EventHandler<T>` and `Callback<Input, Output>` carry the Dioxus runtime, so invoking them from an external watcher restores context needed by runtime-dependent operations.

Event dispatch is synchronous across renderers as of 0.6.0. The `dioxus_prevent_default` attribute is obsolete: call `prevent_default()` conditionally on the event. The call must occur synchronously; after an `.await` it has no effect. LiveView cannot cancel browser behavior from its server-side Rust handler.

The non-HTML `onresize` and `onvisible` handlers install the necessary browser observers automatically; resize data exposes measurements such as `get_border_box_size()`. The event set added `auxclick` and `scrollend` in 0.7.3. File-drop handlers now supply full paths, and Web `WebFileData::file_path` returns `webkitRelativePath` when the browser supplies it. A panic in a Web event handler no longer permanently bricks subsequent interaction as of 0.7.6.

An `onmounted` handler fires after first mount and returns a live element handle for focus, scrolling, and related operations.

## Low-level hook lifecycle

`use_hook` initializes stored state once and retains it until destruction. Use `use_on_destroy` or `use_drop` for cleanup; `use_drop` entered the normal prelude in 0.7.2. `dioxus::core::needs_update()` manually queues the current component when a custom hook changes non-signal state.

```rust
rsx! {
    button {
        onclick: move |_| dioxus::core::needs_update(),
        "Queue a rerender"
    }
}
```

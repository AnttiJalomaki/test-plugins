# UI messages, chat, and streams

## Chat persistence identifiers

Since `4.1.0`, chat identifiers can travel from the client to the server, while
server-assigned response message identifiers travel back to the client. Persist those
server identifiers rather than replacing them with client-only IDs.

`appendResponseMessages` combines prior messages with returned response messages. Use
it to keep the stored conversation aligned with the exact messages generated by the
server, including tool and approval parts needed on a later turn.

An effective persistence flow is:

1. load stored messages and the chat ID;
2. accept the client turn without rewriting existing server message IDs;
3. generate or stream the response;
4. append the response messages with `appendResponseMessages`; and
5. persist the combined sequence atomically.

## Non-blocking custom data streams

`createDataStreamResponse` can write arbitrary data and message annotations before,
during, or alongside an LLM stream. Merge a `streamText` result into the same response
with `mergeIntoDataStream`:

```ts
return createDataStreamResponse({
  execute: async dataStream => {
    dataStream.writeData({
      type: 'source',
      url,
    });

    const result = streamText({
      model,
      messages,
    });

    result.mergeIntoDataStream(dataStream);
  },
});
```

`useChat` can read both the custom data and message annotations. This lets an endpoint
publish sources, status, or other application state without waiting for generation to
finish and without blocking the language stream.

Keep durable message annotations separate from transient progress data when deciding
what to persist.

## Agent-derived UI types

`InferAgentUIMessage<typeof agent>` transfers the agent's tool and output types to UI
components and persistence code:

```ts
export type AgentMessage = InferAgentUIMessage<typeof agent>;
```

On the server, adapt incoming UI messages to the agent and return a UI-compatible
stream with `createAgentUIStreamResponse`:

```ts
return createAgentUIStreamResponse({
  agent,
  uiMessages: messages,
});
```

Use the derived type end to end so tool states, approval parts, and structured output
do not degrade into unvalidated generic message parts during persistence or rendering.

## Direct agent transport

`DirectChatTransport` lets `useChat` invoke an `Agent` directly without building a
separate route transport:

```ts
const { messages, sendMessage } = useChat({
  transport: new DirectChatTransport({ agent }),
});
```

This is useful when the runtime can host the agent directly. A route-based response is
still appropriate when server boundaries, authentication, durable execution, or
deployment topology require it.

Agent chat UI also supports:

- automatic submission of approval responses;
- asynchronous `sendAutomaticallyWhen` conditions; and
- provider metadata preserved across streams and subsequent turns.

Preserve complete response messages before any automatic follow-up, especially for
tool approvals whose replay depends on matching approval IDs.

## Approval UI behavior

Approval is a multi-turn protocol. The first agent or generation call returns a
`tool-approval-request`; it does not remain suspended in memory. The UI should render
the decision, collect approval or denial, append a matching
`tool-approval-response`, and trigger the next call.

An automatic approval-response flow should still enforce the policy that authorized
the action. Do not equate `sendAutomaticallyWhen` with automatic approval: it controls
when a message is sent, not whether a risky action is permitted.

## Browser realtime sessions

Experimental realtime support normalizes direct browser WebSocket sessions across
realtime providers and the AI gateway. It covers:

- server-created ephemeral tokens for browser connection setup;
- audio transcription;
- tools executed by the client;
- normalized realtime events; and
- `UIMessage[]` state shared with ordinary chat UI.

```ts
const realtime = experimental_useRealtime({
  model: realtimeModel,
  api: {
    token: '/api/realtime/setup',
  },
  onToolCall: async ({ toolCall }) => {
    return handleToolCall(toolCall);
  },
});
```

Issue ephemeral credentials from a server endpoint rather than exposing provider
credentials to the browser. Treat client-driven tools as untrusted input, validate
their arguments, and keep sensitive execution on the server even when the realtime
session itself is direct.

## Stream-state checklist

- Consume every generation stream so backpressure does not stall it.
- Handle stream failures from `onError` or in-band error parts.
- Persist server message IDs and complete response messages.
- Distinguish custom transient data from durable message annotations.
- Keep approval request and response IDs paired across calls.
- Preserve provider metadata if a later turn depends on it.
- Use ephemeral browser credentials and validate client tool calls.

## Client paths, URLs, and inputs

### Preserve an exact base URL type

Pass the base URL as the second type argument to `hc`; `$url()` then returns a `TypedURL` with exact protocol, host, and path components (since 4.11.0). This is useful for type-safe keys.

```ts
import { hc } from 'hono/client'

const client = hc<typeof app, 'http://localhost:8787'>(
  'http://localhost:8787/'
)
const url = client.api.posts.$url()
```

### Return only a path

`$path()` applies path and query parameters like `$url()` but returns only the path string (since 4.12.0):

```ts
const path = client.api.posts[':id'].$path({
  param: { id: '123' },
  query: { filter: 'new' },
})
// '/api/posts/123?filter=new'
```

Use it for router inputs and cache keys that must not include an origin.

### Send wire-format strings

Both `param` and `query` values passed to Hono Client must be strings, even when the server-side validator coerces them:

```ts
client.posts[':id'].$get({
  param: { id: '123' },
  query: { page: '1' },
})
```

`hc` interpolates path parameters without URL-encoding them. Encode an ordinary value explicitly. If a raw value is deliberately allowed to span slashes, pair it with a regexp route such as `/posts/:id{.+}`.

```ts
const id = encodeURIComponent('123/456')
await client.posts[':id'].$get({ param: { id } })
```

### Customize query serialization

The `hc` option `buildSearchParams` receives the query object and returns `URLSearchParams` (since 4.11.0). Use it for conventions such as bracketed array keys.

```ts
const client = hc<AppType>('http://localhost', {
  buildSearchParams: (query) => {
    const params = new URLSearchParams()
    for (const [key, value] of Object.entries(query)) {
      if (Array.isArray(value)) {
        value.forEach((item) => params.append(`${key}[]`, item))
      } else if (value !== undefined) {
        params.set(key, value)
      }
    }
    return params
  },
})
```

### Override a single request

Pass fetch initialization as the second call argument:

```ts
client.posts.$post({ json: body }, { init: { signal } })
```

Values in `init` have the highest precedence and can override the method, body, or headers generated by `hc`.

## Response parsing and type composition

### Parse an RPC response by content type

`parseResponse` from `hono/client` accepts an `hc` response promise, parses its body according to `Content-Type`, and throws `DetailedError` when the response is not OK (since 4.9.0).

```ts
import { DetailedError, parseResponse } from 'hono/client'

const result = await parseResponse(client.hello.$get()).catch(
  (error: DetailedError) => console.error(error)
)
```

### Include middleware responses

For routes with multiple handlers, the `hc` route type includes responses returned by middleware or earlier handlers (since 4.10.0). Status checks can narrow every possible response body without a type error.

Pathless handlers also retain the correct application type from 4.11.0. Later 4.11 patches preserve literal unions in validator input inference and merge middleware union response types correctly.

### Add global responses

`ApplyGlobalResponse` adds shared responses from global middleware or `onError()` to every route schema. It is exported from `hono/client` from v4.12.1.

```ts
import type { ApplyGlobalResponse } from 'hono/client'

type ClientApp = ApplyGlobalResponse<
  typeof app,
  { 401: { json: { error: string } }; 500: { json: { error: string } } }
>
const client = hc<ClientApp>('https://api.example.com')
```

Later 4.12 releases export `PickResponseByStatusCode` for selecting a response variant by status.

### Type a custom not-found body

Augment `NotFoundResponse` so `c.notFound()` and Hono Client infer the application's custom 404 body correctly (since 4.11.0):

```ts
import type { TypedResponse } from 'hono'

declare module 'hono' {
  interface NotFoundResponse
    extends Response,
      TypedResponse<{ error: string }, 404, 'json'> {}
}

app.notFound((c) => c.json({ error: 'not found' }, 404))
```

## Validation

### Match the request content type

`validator('json', ...)` and `validator('form', ...)` parse only bodies with the corresponding `Content-Type`. Without that header, the validator receives `{}`, including in `app.request()` tests. For `validator('header', ...)`, read names through lowercase keys such as `value['idempotency-key']`.

### Use Standard Schema

Install `@hono/standard-validator` and use `sValidator()` with any Standard Schema-compatible schema, including Zod, Valibot, and ArkType. Read the validated value with `c.req.valid()`.

```ts
import { sValidator } from '@hono/standard-validator'
import * as z from 'zod'

const schema = z.object({ name: z.string(), age: z.number() })
app.post('/author', sValidator('json', schema), (c) =>
  c.json(c.req.valid('json'))
)
```

`@hono/zod-validator` supports Zod 4 from Hono 4.8.0.

## Testing and large applications

### Pass client options through `testClient`

`testClient` accepts options for its underlying `hc` client (since 4.8.0). Use those options when the tested client needs the same client configuration as production.

### Inject request environment values

Pass a mock environment as the third argument to `app.request()` to populate `c.env`:

```ts
const res = await app.request('/posts', {}, {
  API_HOST: 'example.com',
  DB: mockDb,
})
```

This tests bindings and other runtime values without starting a server.

### Precompile large client types

For a large application, export a compiled client factory so declaration generation performs expensive route-type instantiation instead of every editor session:

```ts
import { hc } from 'hono/client'
import { app } from './app'

export type Client = ReturnType<typeof hc<typeof app>>
export const hcWithType = (...args: Parameters<typeof hc>): Client =>
  hc<typeof app>(...args)
```

Use matching Hono versions in separate client and server projects, and connect them with TypeScript project references.
